## What's Working

1. **Cohesive identity system.** The Orbitron → JetBrains Mono → Inter hierarchy, neon palette, and three-panel layout form a consistent, opinionated language. Every screen belongs to the same product.
2. **Performance consciousness.** Boot bar now uses `transform: scaleX()` instead of width. `will-change` is used judiciously. Semantic z-index scale. The code takes rendering seriously.
3. **Error handling maturity.** Toast system with 4 severity levels, global error capture (`_niuLog`), GitHub API retry logic (3 attempts), rate limit messaging. Most single-file dashboards don't have this level of resilience.

## Priority Issues

### [P1] Color token drift: 60+ undocumented rgba/hex values

**What**: The DESIGN.md palette defines 12 named colors, but the CSS uses ~60 additional distinct rgba values and hex colors not documented anywhere. Most are transparency variants of existing colors (e.g. `rgba(0,255,242,0.12)` is cyan at 12%), but values like `#ffd000`, `#b0b0d0`, `#9999b5`, `#f0f0f8`, and `rgba(120,120,140,.15)` have no token name.

**Why it matters**: DESIGN.md is supposed to be the source of truth. With 60+ undocumented colors, the system has drifted — new contributors can't tell what's intentional and what's one-off. Colors like `#f0f0f8` (a variant of text-primary) appear 4 times across the CSS, suggesting they should be tokens.

**Fix**: Either add all common variants as named tokens in DESIGN.md (e.g. `text-primary-bright: #f0f0f8`; `tag-bg: rgba(80,80,160,0.15)`), or restructure the palette to use CSS `color-mix()` / opacity for variants.

**Suggested command**: `/impeccable document` or `/impeccable harden`

### [P1] No contextual help or documentation (H10 scored 2/4)

**What**: The only help in the entire interface is the keyboard hint bar (`S` search · `1-6` filter · `Esc` close). There are no tooltips, no inline explanations, no docs link, no onboarding.

**Why it matters**: Niu-Dash has complex features (GitHub auto-sync, release management, token setup, ecosystem view, tag-based filtering). A first-time or returning user has no way to learn about these. The GitHub token status indicator shows "🔑 token" with a colored dot but no explanation of what it means or how to set it up.

**Fix**: Add a minimal help system — tooltips on icon buttons, a brief "how to" in the empty state, a link to a README/docs page. At minimum, add `title` attributes to all icon-only buttons and indicators.

**Suggested command**: `/impeccable onboard`

### [P2] Recommendation card side-stripe violates DESIGN.md's own "Don't" rules

**What**: DESIGN.md says "Don't gunakan side-stripe borders (border-left/right > 1px sebagai aksen)" with the exception of the 3px feed card category strip. But `.dp-rec-item` at line 357 uses `border-left:3px solid var(--cyan)` — this is the same pattern the rules explicitly ban.

**Why it matters**: Design systems lose credibility when they break their own rules. If side-stripes are banned except for feed cards, rec cards shouldn't use them.

**Fix**: Replace rec card `border-left:3px` with a full border treatment (2px border on the left side with the priority color, or a background tint with a smaller accent dot/badge).

**Suggested command**: `/impeccable polish`

### [P2] No loading skeleton for feed on initial load

**What**: When the page first loads, the feed panel is empty until `renderFeed()` finishes building the flat list and rendering cards. The empty state (`#no-results`) has `display:none` by default, so the area is just blank.

**Why it matters**: On slow connections or with many projects (105+), the render step has measurable latency. The user stares at a blank white-gray-black void, unsure if the dashboard is working.

**Fix**: Add 3-4 skeleton card placeholders that show during the initial render cycle, or keep the boot overlay visible until feed is fully populated.

**Suggested command**: `/impeccable harden`

### [P3] Font name case mismatch in rendered HTML

## Persona Red Flags

### Alex (Power User)
- **Keyboard shortcuts**: S for search ✓, Esc for close ✓, 1-6 for filter ✓. But no arrow keys to navigate cards in feed — forces mouse interaction for browsing.
- **No bulk actions**: Can't select multiple projects for batch operations. All interactions are one-at-a-time.
- **No customizable layout**: Can't resize panels or hide sections. The 3-panel layout is fixed.
- **Spring animation lag**: The `spring-btn` cubic-bezier adds ~350ms to every button interaction. Alex would feel this as sluggishness on rapid operations.
- **The `refresh-btn` loading state is visual-only** — no status text or countdown. Alex has to guess when the refresh completed.

### Sam (Accessibility)
- **Focus indicators**: Present on all major interactive elements ✓. But `:focus-visible` global rule was recently added — was missing before.
- **ARIA**: Sidebar has `role="navigation"`, detail has `role="complementary"` ✓. Search input has `aria-label` ✓.
- **Color-only meaning**: Status badges ("▶ ACTIVE", "⟐ STAGING", "◌ PAUSED") use color + icon + text — accessible ✓. But category strips on feed cards are color-only (3px left border in green/cyan/amber/magenta/dim) — screen readers get no category info from these.
- **Keyboard navigation**: Feed cards use `tabindex="0"` + `role="button"` — functional but non-standard. A list of `<button>` or `<a>` elements would be more robust.
- **No skip-to-content link**: Keyboard users must tab through the entire sidebar before reaching the feed.

### Project-Specific: Niumination (The Developer/Operator)
- **Profile**: Solo developer managing 105+ projects from Aceh Tengah. Uses this dashboard daily for status checks and release tracking.
- **Behaviors**: Efficiency-obsessed. Looks at the dashboard multiple times a day. Knows all projects by category. Prefers keyboard over mouse.
- **Red Flags**: The filter bar with 12+ options is too much for "quick status check" — needs a faster path to recent/active projects. No way to bookmark or pin frequently-viewed projects. The 3-panel layout can't be collapsed to a 2-panel view for focused work.

## Questions to Consider

1. **Category colors — should they be self-documenting?** Currently green/cyan/amber/magenta/muted require memorization. Would a small inline legend or tooltip on the first hover eliminate the learning curve?
2. **The spring animation — does it serve the user or the aesthetic?** It's documented as intentional, but at 350ms it adds perceptible latency to every button click. For a daily-use tool, would 150ms ease-out serve better?
3. **Font-rajdhani — when does it earn its keep?** It's loaded as a 4th font with zero usage. Either use it for something (version numbers, dense stats) or remove the Google Fonts weight to reduce load.
4. **What would make this dashboard feel like it was made for one specific person?** The dark nexus aesthetic is strong, but personal touches (customizable pinned projects, favorite filters, a "what changed since yesterday" view) would make it feel like Niumination's command center, not a template.
